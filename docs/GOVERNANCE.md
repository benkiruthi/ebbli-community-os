# Governance

Ebbli Community OS is currently maintainer-led while its contributor community and technical foundation are established.

## Current model

The maintainer is responsible for:

- project direction and scope;
- repository administration;
- release decisions;
- security response;
- Code of Conduct enforcement;
- reviewing and merging contributions;
- protecting the distinction between open-source infrastructure and private Ebbli systems.

## Decision-making

Routine decisions may be made through issues and pull requests. Significant changes should include public discussion and written reasoning, especially when they affect:

- security and privacy;
- project licensing;
- governance;
- database compatibility;
- public APIs;
- moderation and trust systems;
- responsible AI boundaries.

Maintainers may decline contributions that conflict with the mission, introduce unsafe defaults, create excessive maintenance burden, or expose proprietary or sensitive systems.

## Path to expanded governance

As the project develops, trusted contributors may become reviewers or maintainers based on sustained, constructive participation and demonstrated responsibility.

Future governance may include:

- module maintainers;
- a technical steering group;
- community and safety advisors;
- transparent release processes;
- documented voting or consensus rules.

## Conflicts of interest

Maintainers and reviewers should disclose material conflicts when evaluating proposals, partnerships, vendors, or competing commercial interests.

## Community voice

Technical contributors are not the only stakeholders. Product decisions should seek input from community organizers, moderators, accessibility advocates, and people using mobile-first services in African contexts.
